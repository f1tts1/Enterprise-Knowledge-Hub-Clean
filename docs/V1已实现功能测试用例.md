# 当前已实现功能与手工测试用例

本文档用于手工验证当前已经落地的业务链路。旧的 F03/F13 阶段说明已经过时，当前能力以代码为准。确定性单元/契约测试、轻量 RAG 回归和故障演练的证据等级统一见 `docs/TESTING.md`；不要把下列步骤“可执行”写成已经实际通过。

## 当前测试范围

- 用户注册、登录、JWT 鉴权、当前用户信息。
- 个人知识库创建、列表、详情、删除。
- 文档上传到 MinIO、document 表落库、indexing_task 表创建。
- RabbitMQ 异步触发 Java 消费者调用 Python。
- Python 从 MinIO 下载文件并解析 TXT、Markdown、PDF、DOCX。
- Python 切分 chunk，生成本地 embedding，并写入 Qdrant。
- Java 校验 Python 返回 `INDEXED` 后更新索引状态。
- Java 检索接口校验知识库归属。
- Python 检索接口使用 `ownerUserId + kbId` 过滤 Qdrant。
- 不同用户只能检索自己的知识库内容。
- Java 复用权限过滤后的检索结果，调用 Python 完成最小同步 RAG 问答。
- RAG 返回结构化 `ANSWERED`、`NO_CONTEXT` 或 `INSUFFICIENT_CONTEXT`；`citations` 只包含答案实际使用的 chunk，`retrievedChunks` 保留候选命中。
- 删除文档时先进入 `DELETING` 并对业务不可见，再清理 Qdrant vectors 和 MinIO object，检索不再返回该文档 chunk。

当前不测试：

- SSE 流式输出。
- 多轮会话落库、反馈 API。
- Agent 工作流。
- hybrid search、rerank、query rewrite 和正式 RAG 评测平台。
- Compose 镜像构建/启动；本轮只提供本机配置，实际验证边界见 `docs/DEPLOYMENT.md`。

这些未进入当前实现范围，不应该写进“已完成”或伪造测试证据。

## 推荐一键测试：权限检索隔离

```bash
./scripts/test_retrieval_permission.sh
```

脚本会执行：

1. 注册 Alice 和 Bob。
2. 分别登录并创建各自知识库。
3. 上传 Alice 的 RabbitMQ/Qdrant 文档。
4. 上传 Bob 的财务/HR 文档。
5. 等待每份文档索引进入 `INDEXED/SUCCESS`。
6. Alice 检索自己的知识库，应命中 Alice 文档。
7. Alice 用 Bob 的关键词检索自己的知识库，不应命中 Bob 私有内容。
8. Bob 检索自己的知识库，应命中 Bob 文档。
9. 删除 Alice 的一份 Qdrant 文档，再检索同一关键词时不应返回该文档 chunk。
10. Alice 直接检索 Bob 的 `kbId` 应返回 404。
11. Bob 直接检索 Alice 的 `kbId` 应返回 404。

如果 Java 后端不是 8080：

```bash
BASE_URL=http://localhost:8081 ./scripts/test_retrieval_permission.sh
```

## 手工最小演示路径

### 1. 注册

```bash
curl -X POST "http://localhost:8080/api/v1/auth/register" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "alice",
    "password": "Password123",
    "nickname": "Alice",
    "email": "alice@example.com"
  }'
```

预期：

- HTTP 200
- `success=true`
- 返回用户 id，不返回 passwordHash

### 2. 登录

```bash
curl -X POST "http://localhost:8080/api/v1/auth/login" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "alice",
    "password": "Password123"
  }'
```

预期：

- HTTP 200
- `data.accessToken` 非空
- 后续请求使用 `Authorization: Bearer <accessToken>`

### 3. 创建知识库

```bash
curl -X POST "http://localhost:8080/api/v1/knowledge-bases" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Java 后端知识库",
    "description": "用于测试文档索引和向量检索"
  }'
```

预期：

- HTTP 200
- `visibility=PRIVATE`
- `status=ACTIVE`

### 4. 上传 TXT 文档

```bash
printf "RabbitMQ 用于文档上传后的异步索引任务。Qdrant 保存 chunk 向量和权限 payload。" > /tmp/ekb-test.txt

curl -X POST "http://localhost:8080/api/v1/knowledge-bases/{kbId}/documents" \
  -H "Authorization: Bearer {token}" \
  -F "file=@/tmp/ekb-test.txt"
```

预期：

- HTTP 200
- `indexStatus=PENDING_INDEX`
- `indexingTaskId` 非空
- MinIO 中存在 object
- MySQL 中存在 `document` 和 `indexing_task`

### 5. 轮询索引状态

```bash
curl "http://localhost:8080/api/v1/documents/{documentId}/index-status" \
  -H "Authorization: Bearer {token}"
```

预期最终状态：

- `documentIndexStatus=INDEXED`
- `taskStatus=SUCCESS`
- 初次上传时 `attemptNo=0`、`triggerType=UPLOAD`
- `chunkCount > 0`

如果模型缺失、FastAPI 未启动、Qdrant 不可用或 MinIO 配置错误，文档状态会进入 `INDEX_FAILED`，任务状态会进入 `FAILED`，错误信息写入 `documentErrorMessage` 或 `taskErrorMessage`。可调用 `POST /api/v1/documents/{documentId}/index-retry` 人工重试；重试后应得到新的 `indexingTaskId`、递增的 `attemptNo` 和 `triggerType=MANUAL_RETRY`，旧 task 仍保持 FAILED。

### 6. 检索知识库

```bash
curl -X POST "http://localhost:8080/api/v1/knowledge-bases/{kbId}/retrieval/search" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "RabbitMQ 和 Qdrant 的作用",
    "topK": 5
  }'
```

预期：

- HTTP 200
- `success=true`
- `data.vectorStore=qdrant`
- `data.vectorCollection=rag_chunks_v1`
- `data.records` 返回 chunk 命中
- 每条记录包含 `score`、`docId`、`chunkId`、`fileName`、`pageNo`、`chunkIndex`、`text`

### 7. 删除文档并验证 vectors 清理

```bash
curl -X DELETE "http://localhost:8080/api/v1/documents/{documentId}" \
  -H "Authorization: Bearer {token}"

curl -X POST "http://localhost:8080/api/v1/knowledge-bases/{kbId}/retrieval/search" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "RabbitMQ 和 Qdrant 的作用",
    "topK": 5
  }'
```

预期：

- 删除接口 HTTP 200，`success=true`
- Java 会同步调用 Python 清理该文档在 Qdrant 中的 vectors，并删除 MinIO object
- 后续检索结果不再包含被删除文档的 chunk

### 8. 验证最小同步 RAG 与引用语义

```bash
curl -X POST "http://localhost:8080/api/v1/knowledge-bases/{kbId}/rag/ask" \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "question": "RabbitMQ 和 Qdrant 分别承担什么职责？",
    "topK": 5
  }'
```

存在足够证据时预期：

- `answerStatus=ANSWERED`、`noAnswer=false`、`noAnswerReason=null`。
- 答案正文至少包含一个合法 `[片段 n]`。
- `citations` 只映射正文实际引用的编号；不能因为 `retrievedChunks` 返回 5 条就伪造 5 条 citation。

无检索上下文时预期 `answerStatus=NO_CONTEXT`；模型判断上下文不足或启用的相关性阈值过滤掉全部候选时预期 `answerStatus=INSUFFICIENT_CONTEXT`。两种 no-answer 均要求 `noAnswer=true`、原因非空且 `citations=[]`。当前阈值默认关闭，未经当前数据校准不应任意启用。

对应脚本：

```bash
./scripts/test_rag_ask.sh
./scripts/test_rag_empty_kb.sh
./scripts/test_rag_permission.sh
```

正常回答和权限脚本需要真实 LLM 配置，不纳入确定性核心 CI。

## 当前容易暴露问题的地方

- MySQL 没执行建表 SQL：注册或创建知识库会失败。
- SQL 会删除数据库：不要在有重要数据的库上直接执行。
- Redis 没启动或密码不一致：认证缓存、refresh token 或知识库访问缓存会降级或失败。
- RabbitMQ 没启动或账号不一致：上传后的索引任务会停留在 `PENDING_INDEX/PENDING`，恢复 RabbitMQ 后可通过重投器或 `/index-retry` 投递。
- MinIO 没启动：文档上传会返回存储服务错误。
- MinIO/RabbitMQ/Redis 凭证必须与本机配置一致；仓库只提供占位模板，不提供可复用的真实默认凭证。
- Qdrant 没启动或被系统代理影响：Python 写入或检索会返回 502。
- 本地 embedding 模型没下载：Python 索引和检索会失败。
- 上传重复文件：会被 SHA-256 去重拦截，这是预期行为。
- Python 服务没启动：RabbitMQ 消费者会拿到任务，但调用 Python 失败，`document.index_status` 会进入 `INDEX_FAILED`，`indexing_task.status` 会进入 `FAILED`。
- Qdrant 或 MinIO 删除失败：文档会进入 `DELETE_FAILED` 并保持 `is_deleted=1`，不会重新出现在列表或检索结果中；再次调用删除接口会认领新的 `delete_generation`，旧删除调用不能覆盖新状态。
